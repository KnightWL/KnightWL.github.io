runtime探究（二）：方法交换

使用场景

现在有这么一个情况：
一个已经开发了N年，非常庞大的项目，突然间产品经理说：“咱们要给app加上一些数据采集功能，要求采集到用户每次打开一个我们的页面的动作”。

我们该如何填上这个坑呢？如果使用 常规手段，有这样2种大概可行的办法：
  • 写一个UIViewController的子类重写ViewDidLoad方法的实现，然后将该子类作为所以app中controller的基类。
  • 给UIViewController写一个 category，在category中重写ViewDidLoad，在pch文件中import进来 ；
首先我们来分析第一种：
如果本身 *app中存在这样的基类而且所有的controller都是其子类*，那这时候我们使用该方法是非常方便可行的，但是一旦没有这样的基类或者说这样的基类可能有多个，每个子类都继承子不同的基类，那我想这着实是一场编程灾难。
接下来看第二种：
首先这个想法肯定是比上一个靠谱的，毕竟不管有没有上述那么一个基类，我们都能通过在pch中引入category使所有的 *__kindof UIViewController\**的ViewDidLod方法实现变成我们在category中重写的实现，但是可行吗？category中实现方法与我们在类的.m文件中有一个非常大的区别： __category中不能使用super方法__，也就意味着我们在category中实现ViewDidLoad的重写时无法调用这句 *[super viewDidLoad]*，我想这个灾难应该比上一种还可怕。所以现在我们对category有多了一层认知： __最好不要使用category来重写系统方法的实现。__

其实也引出了方法交换的使用场景： 全局低耦合性修改系统方法实现。

使用方法

这里有一个很巧妙的思路可以在耦合性非常低的情况下来实现全局的系统方法重写，步骤如下：
  • 创建一对category文件
  • 在category的.m中引入 \<objc/message.h>
  • 在.m中实现一个自定义的方法 DIY_viewDidLod，在其实加入自己要添加的方法实现然后在该方法内调用 [self DIY_viewDidLoad]
  • 在 load方法中使用 runtime将 viewDidLoad与 DIY_viewDidLoad的方法实现互换。（ 最好在互换时将互换过程写在dispatch_once的block中，虽然我们都知道load方法只在本文件加载进ram时调用一次，但是我还真的碰到过一个bug对一个文件的load方法执行过多次的情况）
这里边其实存在一些需要讨论的东西：

  • 为什么使用 load方法？有没有可以代替的方法？
    _因为load方法是在文件被加载进内存时就调用的方法，能最大限度保证全局的viewDidLoad方法都被我们所自定义的方法实现替换； initialize方法其实也可以实现我们的需求，但是该方法不止会在加载进内存时调用，而是在整个运行过程中会存在多次调用，不过仍然可以通过dispatch_once来实现我们的需求，其实在swift中因为没有load方法，也确实是通过initialize+dispatch_once实现的。_

  • 为什么在DIY_viewDidLoad中又调用了DIY_viewDidLoad，难道不会变成死循环吗？
    因为我们已经在load中将DIY_viewDidLoad和原生的viewDidLoad的方法实现做了交换，所以此时在DIY_viewDidLoad的实现中调用的 [self DIY_viewDidLoad]其实调用的是[self viewDidLoad]的实现，所以不存在死循环，反而如果在DIY_viewDidLoad中调用viewDidLoad会产生死循环。

代码

#import "UIViewController+ExchangeViewDidLoad.h"
#import <objc/message.h>
@implementation UIViewController (ExchangeViewDidLoad)
+ (void)load{
     static dispatch_once_t onceToken;
    dispatch_once(&onceToken, ^{
        Method met1 = class_getInstanceMethod(self, @selector(viewDidLoad));
        Method met2 = class_getInstanceMethod(self, @selector(wl_viewDidLoad));
        method_exchangeImplementations(met1, met2);
    });
}

- (void)wl_viewDidLoad{
    NSLog(@"交换成功");
    [self wl_viewDidLoad];
}
@end

JS中的继承

JS中的对象类型

JS中的对象一共有两个大类型:

  • Object对象,即普通对象
  • Function对象,即函数对象

二者的相同点

  • 每个函数都有__proto__属性

二者的区别

  • 与普通对象不同,函数对象的创建是通过new Function() 的方式创建的
  • 函数对象有一个专属的属性prototype指向该函数对象的原型对象,是所有普通对象不具有的.
  • 所有通过某构造函数A所创建的对象都具有一个constructor属性,该属性的指针指向构造函数对象A

原型对象

  • 原型对象（Person.prototype）是 构造函数（Person）的一个实例;

  • 原型对象其实就是普通对象（但 Function.prototype 除外，它是函数对象，但它很特殊，他没有prototype属性（前面说道函数对象都有prototype属性））。

  • 远行对象的主要作用是用于继承

__proto__属性

所有对象在创建的时候都会有一个__proto__属性,该属性指向 对象的构造函数的prototype

模式一:原型继承

原型继承是让父类对象作为子类对象的原型,从而达到继承的目的,分为两种情况

  • 将一个父类对象作为子类对象的原型

  • 直接将父类对象的原型作为子类对象的原型
    ` function object(parent){
    function sub(){

    }
    sub.prototype = parent;
    return new sub();
    }

runtime探究（一）

自从知道了runtime这个东西后，就一直爱不释手，一是用起来比较炫酷，非常适合装逼；第二是因为runtime作为OC的底层实现，多了解、使用能让我们更清楚知道OC的运作原理，并且runtime还这么强大，不用那岂不是暴殄天物。

本着复习runtime的精神，打算将整个探究过程分为三个篇章：第一篇，先不讲原理，先说runtime的一些低级用法诸如关联属性、获取属性列表、方法列表、变量列表、协议列表等；第二篇，扯一扯一些原理性的东西诸如isa、class、方法传递；第三篇再讲一些AOP方向的高级应用。
为啥不先说原理呢？常言道：授人以鱼不如授人以渔，但是我要是想先让大家吃条鱼再给你们鱼竿肯定效果更好啊，直接给鱼竿万一鱼没上钩先饿死了呢？

当然，想在我们的代码中使用runtime是有前提的，前提就是:#import <Objc/runtime.h>

一、关联属性

听起来有点吓人，看着就像很专业的东西。其实简单的很，就是在程序运行期间为某类对象、实例对象动态添加一个属性。
用法：
associate setter方法： void objc_setAssociatedObject(id object, const void *key, id value, objc_AssociationPolicy policy)
参数：object 要绑定属性的对象
key 要绑定的属性的key
id 要绑定的属性
policy 绑定属性时使用的内存管理方式
（
OBJC_ASSOCIATION_ASSIGN = 0, ==asign
OBJC_ASSOCIATION_RETAIN_NONATOMIC = 1, ==nonatomic,strong
OBJC_ASSOCIATION_COPY_NONATOMIC = 3, ==nonatomic,copy
OBJC_ASSOCIATION_RETAIN = 01401, ==strong
OBJC_ASSOCIATION_COPY = 01403 ==copy
）

associate getter方法： id objc_getAssociatedObject(id object, const void *key)
参数：object 已经绑定属性的对象
key 绑定的key
很明显，两个C语言的API，用起来也是相当简单。

使用了setter方法，就等于给该对象动态添加了一个值为value，键为key的属性，常常用在用字符串给UIView做标识来区分（类似于tag），或者给alert、action、 button等绑定一个处理事件的block（我在我之前的文章alert、action的一些拓展中提到过）。

需要注意的一点是设置了关联属性并不会像我们正常添加的属性一样自动生成setter方法和getter方法，只能使用associate setter/getter方法来对值进行设置和获取。而且也不会被添加到类的属性列表和变量列表。

遍历类的各种列表

runtime作为OC的底层，广义上说其实可以算作元编程，顺便普及一下，有时候总能看到什么元类、元编程之类的词汇，觉得好高深，其实所谓元xx，可以认为是关于xx的xx，元编程就是关于编程的编程。既然是关于编程的编程，那么在我们无法用代码通过常规手段知道一个类的属性、变量、方法、遵守的协议都有些什么的时候，runtime就又要立功了。

NSTimer倒计时那些事

以前使用NSTimer很少，也没怎么深入探究过，最近使用instruments跑了一遍以前的项目才发现，在timer这我给自己埋了很多坑。

在我过去的项目中发现，凡是持有timer的页面（例如许许多多接收验证码的页面），在该dealloc的时候都没有正常执行的dealloc，导致了不少内存泄漏。
以前总是习惯在viewController的dealloc方法中来执行

上文已经说了，dealloc是不会执行的，那么这是为什么呢？

原来，只要timer在初始化时，传入的repeat属性是YES，那么这时候timer就会强引用到传入的target（一般是我们的controller），那么事情很简单了，这二位仁兄retainCycle了。controller就算被pop出navigation栈或者被dismiss掉，但是还有timer引用它，所以它肯定不会dealloc了，在dealloc中去invalidate我们的timer明显也是不行了。

那么许多胖友可能会想了，那我干脆controller弱引用timer，或者不持有timer这个属性不就行了，这个想法啊，too young， sometimes naive🤓你会发现controller还是不会dealloc，why？

timer如果不invalidate掉的话会一直保存在内存中，那么对应的controller也一直在了。

那怎么办呢？难道就不能用timer做倒计时了？

非也，也不是不可以的。一个比较折中的方法，在timer自己的selector中一旦倒计时到0就invalidate然后置空，下次再开启倒计时就重新创建一个新的timer。
不过这样会出现一个非常有意思的情况，你会发现pop或者dismiss掉controller之前，如果timer的倒计时没有结束，那么controller不会第一时间dealloc，而是会在timer倒计时结束后再dealloc。

在使用alert、action时总觉得特别难用，instance的声明创建与点击事件实现完全分开不说，点击事件的代码还全部都要放到代理方法里，而且alert、action的index还可能存在很多，那么一个alert、action的点击事件就已经相当复杂，多个它们的点击事件放到一个代理方法里想想就头疼。

iOS可变参数问题

之前在做的项目有这么一个需求：app不允许多端登录，多端登录互踢。我们的解决思路是直接appdelegate监测从融云服务器发来的状态切换通知和我们自己服务端做请求时返回的statusCode，一旦发现被踢，那么直接在window上present出一个登录界面。但是这样存在一个问题，那就是如果present之前界面上存在alert、actionsheet两种窗口，那么就算登录界面被模态出了，alert和actionsheet仍然存在，并显示在登录界面之上。

（1）首先在函数里定义一具VA_LIST型的变量，这个变量是指向参数的指针；
（2）然后用VA_START宏初始化刚定义的VA_LIST变量；
（3）然后用VA_ARG返回可变的参数，VA_ARG的第二个参数是你要返回的参数的类型（如果函数有多个可变参数的，依次调用VA_ARG获取各个参数）；
（4）最后用VA_END宏结束可变参数的获取。

AFNetworking 内存泄漏问题

前几天突心血来潮用instruments跑了跑之前从同事手里接过来的项目，结果一跑就让我抓到了个AFN的问题。

每向后台发出一次请求，就通过instruments截获到一个内存泄漏，而且每次请求发出后发生内存泄漏的call tree和retain circle都是一样的如下2图：


那么问题已经非常明显了，我特意去查了下NSURLSession的delegate持有方式为strong，而AFURLManager及其子类对象对NSURLSession对象为强引用，且恰恰AFURLManager及其子类对象遵守了NSURLSessionDelegate，所以造成了循环引用，请求结束后NSURLSession对象与AFURLManager对象并没有得到释放，造成了内存泄漏。
特意去调研了一下这个问题应该是session的ARC机制下的历史遗留问题，那么既然如此如何去解决呢？
两种方法：
  • 将session写成单例，将AFURLManager对session的引用改成weak
  • 将manager写成单例

    第一种方法非常好理解，既然是session的历史遗留问题，那么直接从session下手将其改为单例之后session常驻内存不用担心被释放掉，所以session的生命周期长于持有它的manager，可以放心的将manager对session的引用改为weak；
    第二种看上去有点舍近求远，明明是session的问题，改manager有什么用的？点进afn的源码就会发现，session的创建都是在manager中进行的，而且manager内部不存在修改session的代码，那么将manager封装成单例的话，session也就相应的变成了单例，虽然它们之间的循环引用还存在，但是这已经不会造成任何内存泄漏问题了–单例的生命周期等同于app生命周期。

对比一下两种解决办法的优劣：
  • 第一种简单容易理解，错误出在哪里就去哪里解决，但是因为session是在manager内部创建的，而且我们要将session改成单例还需要去改manager对session的强引用，就造成了对AFN内部代码的大量修改。
  • 我个人更推荐第二种算法，因为它看似舍近求远，但是其实是从最根源解决了AFN的内存泄漏问题， 并且manager的创建一般都是在我们对AFN的二次封装或者直接使用时进行的，所以不需要修改AFN内部代码，工作量也小于第一种。
